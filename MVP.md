# MVP — Сервис дистрибуции научных статей

## 1. Ценностное предложение

**Для кого:**  
Для учёных и научных сотрудников университетов (целевой сегмент), которые обязаны ежегодно публиковаться и тратят значительное время на поиск релевантных научных статей.

**В какой ситуации:**  
При подготовке новых публикаций, написании обзоров литературы, отслеживании актуальных исследований в своей области. Пользователь подписан на рассылку arXiv или Google Scholar Alerts, но ежедневно получает десятки статей, большинство из которых не относятся к его узкой тематике.

**Что мы обеспечиваем:**  
Мы сокращаем время на поиск релевантных статей с **2–3 часов в неделю до 30 минут**, а также даём **почти гарантированное обнаружение ключевых публикаций** (от выбранных авторов, по заданным ключевым словам и темам) благодаря персонализированной подборке, обучающейся на предпочтениях пользователя.

**Измеримый эффект:**  
- Снижение затрачиваемого времени на поиск статей в 4–6 раз (отслеживается через опросы пользователей после 2 недель использования).  
- Доля пользователей, которые после 2 недель подтверждают, что «стали чаще и быстрее находить релевантные статьи» — не менее 80%.

**Условия получения ценности:**  
1. Пользователь регистрируется и настраивает предпочтения (тематические области, авторы, ключевые слова).  
2. Пользователь получает ежедневную подборку из **15 статей** в Telegram-боте.  
3. Пользователь ставит лайки/дизлайки под каждой статьёй — это данные для обучения алгоритма.  
4. В течение **первых 2 недель** алгоритм адаптируется и начинает выдавать преимущественно релевантный контент.

## 2. Структура функциональности решения

### Ядро MVP (обязательно к реализации в ближайшие 2 недели)

| Функция | Описание |
|--------|----------|
| **Регистрация / вход** | По email + пароль, без подтверждения email (для упрощения). |
| **Настройка предпочтений** | Выбор тематических областей (из списка arXiv или заранее подготовленного), добавление авторов (по фамилии или ID в Google Scholar), ключевых слов (с возможностью комбинировать AND/OR). |
| **Telegram-бот** | Единственный канал доставки. Пользователь привязывает аккаунт после регистрации (через команду /start и ввод логина/пароля). |
| **Ежедневная рассылка** | 15 статей, присылаемых одной пачкой (или поочерёдно) каждый день в заданное время. Каждая статья приходит отдельным сообщением с кнопками «👍» / «👎». |
| **Оценка статей** | Лайк/дизлайк. После оценки бот может ответить подтверждением («Оценка сохранена»). Возможность отменить/изменить оценку в течение суток (кнопка «Отменить»). |
| **Обратная связь после подборки** | После отправки всех статей бот спрашивает: «Насколько вам понравилась сегодняшняя подборка?» (оценка от 1 до 5). Это поможет собирать метрику удовлетворённости. |
| **Редактирование предпочтений** | Пользователь может изменить темы, авторов, ключевые слова в любой момент (через меню бота). При смене темы — предупреждение, что обучение начнётся заново. |
| **База статей** | Источники: arXiv, bioRxiv, открытые журналы (например, PLOS ONE). Парсинг метаданных (название, авторы, аннотация, ссылка) через API или веб-скрапинг. |
| **Простой алгоритм ранжирования** | На основе выбранных тем, авторов, ключевых слов и истории лайков/дизлайков. Первые 2 недели — комбинация: 50% от настроек + 50% от коллаборативной фильтрации. |

### Поддерживающие функции (внутренние для MVP)

- Логирование всех действий пользователя (регистрация, настройки, оценки).
- Админ-панель для мониторинга количества пользователей, ошибок парсинга, просмотра логов.
- Резервное копирование данных раз в сутки.

### Отложенные функции (post-MVP)

| Функция | Причина отложения |
|--------|-------------------|
| Анализ полных текстов | Требует значительных вычислительных мощностей и юридического согласования. |
| Экспорт в Zotero/Mendeley | Не критично для проверки гипотезы ценности. |
| Корпоративные аккаунты | Необходимость в B2B-сегменте, который пока не в фокусе. |
| Мобильное приложение | Достаточно Telegram-бота для MVP. |
| Оценка по шкале 1–5 | Усложняет интерфейс; лайк/дизлайк даёт достаточно данных для первого алгоритма. |
| Email-рассылка | Дополнительный канал, можно добавить позже. |

## 3. Пользовательские сценарии

### Основной сценарий (Happy Path)

1. **Обнаружение сервиса** – через сарафанное радио или нашу рекламу.
2. **Регистрация** – пользователь переходит по ссылке на веб-страницу (или сразу в боте), вводит email и пароль.
3. **Настройка** – бот задаёт вопросы:  
   - «Выберите одну или несколько научных областей» (список).  
   - «Введите авторов, за которыми хотите следить» (опционально).  
   - «Введите ключевые слова» (опционально).  
4. **Получение подборки** – бот отправляет первую подборку из 15 статей на следующий день в выбранное время (по умолчанию 9:00, но можно изменить).  
5. **Взаимодействие** – пользователь читает статьи, под каждой ставит 👍 или 👎.  
6. **Обратная связь** – после подборки бот спрашивает оценку от 1 до 5.  
7. **Улучшение рекомендаций** – через несколько дней пользователь замечает, что статьи стали более релевантными.  
8. **Изменение настроек** – пользователь через меню бота может добавить новых авторов или ключевые слова.

### Альтернативный сценарий (изменение настроек)

1. Пользователь уже зарегистрирован и получает подборки.  
2. Решил сменить тему (например, перешёл в другую область исследований).  
3. В меню бота выбирает «Изменить настройки», меняет тему.  
4. Бот предупреждает: «Внимание! Смена темы приведёт к сбросу накопленных данных обучения. Алгоритм будет перенастроен под новую тему. Продолжить?»  
5. После подтверждения предпочтения обновляются, пользователь начинает получать статьи по новой теме.  
6. Бот сообщает, что для точных рекомендаций потребуется около 2 недель оценок.

### Сценарий отказа (пользователь не ставит оценки)

1. Пользователь зарегистрировался, получает подборки, но не ставит лайки/дизлайки в течение 3 дней.  
2. Бот отправляет напоминание: «Я заметил, что вы не оцениваете статьи. Чтобы алгоритм мог подбирать для вас лучшие материалы, пожалуйста, ставьте 👍 или 👎. Это займёт всего пару секунд».  
3. Если пользователь продолжает игнорировать, через 7 дней бот отправляет повторное сообщение с объяснением, что без оценок персонализация не работает.  
4. Если пользователь всё равно не оценивает, он продолжает получать подборки на основе начальных настроек (без обучения).

## 4. Функциональные требования и критерии приемки

### User Story 1: Регистрация
*Как пользователь, я хочу зарегистрироваться в сервисе, чтобы получать персонализированные подборки статей.*

**Критерии приемки:**
- Пользователь может зарегистрироваться через веб-форму, указав email и пароль (минимальные требования: email валиден, пароль не менее 6 символов).
- После регистрации создаётся запись в базе данных.
- Пользователь автоматически перенаправляется в Telegram-бот для привязки аккаунта (или может привязаться позже по команде /start).
- Бот принимает команду /start, запрашивает email и пароль, проверяет их, связывает Telegram ID с учётной записью.

### User Story 2: Настройка предпочтений
*Как пользователь, я хочу выбрать тематические области, авторов и ключевые слова, чтобы получать статьи, соответствующие моим интересам.*

**Критерии приемки:**
- Пользователь может выбрать одну или несколько областей из предопределённого списка (например, Computer Science, Physics, Biology).
- Пользователь может добавить авторов (ввод текстом, разделение запятой). Система сохраняет их как строки без проверки.
- Пользователь может добавить ключевые слова (ввод текстом, разделение запятой). Система сохраняет их.
- Настройки можно изменить в любой момент через меню бота.
- При изменении темы бот выводит предупреждение о сбросе обучения и требует подтверждения.

### User Story 3: Получение подборки
*Как пользователь, я хочу ежедневно получать подборку статей в Telegram-боте, чтобы быстро знакомиться с новыми публикациями.*

**Критерии приемки:**
- Каждый день в заданное время (например, 9:00) бот отправляет ровно 15 статей.
- Каждая статья приходит отдельным сообщением с заголовком, авторами, аннотацией (обрезанной до 300 символов) и ссылкой на полный текст.
- Под сообщением есть кнопки «👍» и «👎» для оценки.
- Если на момент формирования подборки недостаточно статей по заданным критериям, бот дополняет подборку статьями из смежных тем (или показывает сообщение об отсутствии статей).

### User Story 4: Оценка статей
*Как пользователь, я хочу ставить лайки/дизлайки на статьи, чтобы алгоритм учился моим предпочтениям.*

**Критерии приемки:**
- При нажатии «👍» или «👎» бот сохраняет оценку и отправляет подтверждение («Оценка сохранена»).
- Пользователь может отменить или изменить оценку в течение 24 часов (кнопка «Отменить» рядом с оценкой).
- Оценки сохраняются в базе данных вместе с идентификатором статьи и пользователя.
- Алгоритм использует эти данные для корректировки последующих подборок.

### User Story 5: Обратная связь о подборке
*Как пользователь, я хочу оценить качество всей подборки, чтобы сервис понимал, насколько я доволен.*

**Критерии приемки:**
- После отправки всех статей бот задаёт вопрос: «Насколько вам понравилась сегодняшняя подборка?» с кнопками 1–5.
- Оценка сохраняется в базе данных и может быть использована для метрик качества.
- Если пользователь не ответил, бот не ждёт ответа, просто завершает сессию.

### User Story 6: Изменение настроек
*Как пользователь, я хочу в любой момент изменить выбранные темы, авторов и ключевые слова, чтобы адаптировать подборки под новые интересы.*

**Критерии приемки:**
- В меню бота есть пункт «Изменить настройки».
- Пользователь может последовательно изменить темы, авторов, ключевые слова.
- При изменении любой настройки система обновляет профиль в базе данных.
- Если изменяется тема, бот выводит предупреждение о сбросе истории обучения и просит подтвердить изменение.
- После подтверждения история оценок для этой темы сбрасывается (но не удаляется полностью, а помечается как неактуальная).

## 5. Нефункциональные требования и ограничения

### Производительность
- Подборка из 15 статей должна формироваться и отправляться в течение 5 минут после запуска ежедневного задания.
- Время отклика бота на команду пользователя — не более 2 секунд.
- Система должна выдерживать одновременную работу до 100 активных пользователей (на старте достаточно).

### Надёжность
- Парсинг источников (arXiv, bioRxiv) должен быть устойчив к изменениям структуры страниц. В случае ошибки — повтор через 1 час.
- При недоступности источника — использовать кэшированные данные за последние 3 дня.

### Безопасность
- Пароли пользователей хранятся в хэшированном виде (bcrypt или аналоги).
- Доступ к базе данных только из внутренней сети (или через защищённое соединение).
- Telegram-бот использует только токен, не хранит секреты в коде.
- Администраторский доступ к логам — по отдельному паролю.

### Доступность
- Бот работает 24/7, время недоступности не более 1 часа в сутки (на обновления).
- Поддержка всех платформ, на которых работает Telegram (Windows, macOS, iOS, Android, веб).

### Интеграции
- **arXiv API** — для получения метаданных статей (предпочтительно использовать официальный API, если позволяет).  
- **Google Scholar** — парсинг страниц, но с осторожностью (юридические риски). Для MVP можно ограничиться arXiv и открытыми журналами.  
- **База данных** — PostgreSQL или MongoDB (выбор по удобству разработчика).

### Технические ограничения
- Бюджет на облачные ресурсы (сервер, база данных) — не более 5000 руб./мес на старте.
- Ограничения по API: arXiv — 1 запрос в 3 секунды (для парсинга новых статей использовать cron с разнесением).
- Объём базы данных на старте: ~1000 статей в день, что требует около 10 ГБ за месяц (с учётом аннотаций).

## 6. Риски, связанные с реализацией

| Риск | Мера снижения |
|------|----------------|
| Сложность алгоритма и нехватка данных для обучения | Начать с rule-based (ключевые слова + авторы) и добавить простую коллаборативную фильтрацию после накопления 50+ оценок на пользователя. |
| Юридические вопросы с парсингом | Использовать только открытые источники (arXiv, bioRxiv, PLOS ONE) и официальные API. Избегать парсинга Google Scholar. |
| Пользователи не ставят оценки | Добавить напоминания и объяснение важности оценок; в крайнем случае — продолжать выдачу по настройкам без обучения. |
| Telegram-бот заблокирован / ограничения API | Использовать несколько ботов с ротацией или заранее предусмотреть fallback на веб-интерфейс (но в MVP не требуется). |
